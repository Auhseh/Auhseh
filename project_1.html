Reducing Friction in password reset flows (Quidax)


TL;DR

I attempted to reduce friction in Quidax' password reset flow by introducing a 'Guest Sign In' feature that leverages Security questions and Device logs when OTPs via email fail to deliver.
This reduces drop-offs in the password reset flow and cuts down the number of steps.






Background

Recently, I needed to purchase some crypto-currency (ETH), I had opened an account earlier with one of the leading exchanges in Nigeria, but I couldn’t remember the password to the account, the ‘Sign In’ page had a ‘Forgot password’ option so I figured it would be as easy as resetting my password, I followed through and the flow had 5 steps:

The flow broke when it got to the second Authentication stage (the page prompts that the user is signing in with a new device and requests that the user enters an OTP that has been sent to the user’s email)

Problem is, I never received the email, I tried signing in a couple more times with no success, so I took my business elsewhere, but it bothered me and I wondered how many users they have lost/lose because of this.

The following takes you through how I articulated and attempted to solve the problem.




Problem Statement

I already had a first-hand encounter with the problem so I used the HMW (How Might We) method to properly assimilate it and better troubleshoot for a solution.

Challenge: Redesign the password reset experience

POV: User needs to urgently purchase some crypto-currency, but can’t remember her/his password, hence can’t access the account.

  • “How might we reduce the number of steps needed to reset the password?”
  • “How might we make the password reset process as enjoyable as possible?”
  • "How might we ensure that a user completes the process and avoid drop-offs?"

There were pertinent questions I also needed to answer that would form the foundation for my solution:

  • What was the current state of password resets?
  • What was the industry standard?
  • What were the technicalities behind password resets?
  • What were the tradeoffs to offering the best UX?



Research and Exploration

In attempting to solve this problem, I knew I had to tackle the challenge both from a design and technical perspective.

I started by crawling research papers, blog posts, customer care forums etc. for insights and my biggest discovery here was that there was always a tradeoff between Security and UX, However smooth the UX was, it had to be balanced with the right Security measures. There were essentially 3 factors to consider from a UX perspective:

  • Steps to completion: Number of steps/pages a user has to go through to complete the process
  • Information costs: Number of fields (basically, how much information a user has to enter to complete the process).
  • Effort investments: Number of decisions/additional activities a user has to make/perform to complete the process.

Redesigning

From my research, I discovered that the ‘Sign up’ stage was indeed the best stage to implement security measures. Contingencies always occur, and even the best email delivery systems fail to deliver sometimes, the essence of providing a great UX is providing layers of support and buffers when contingencies occur, and the ‘Sign Up’ stage turned out to be the best place to implement these to address situations like;

  • Cases where the user no longer has access to the email used at sign up
  • Cases where the user does not receive the verification link etc.

My focus on this project was the Password Reset Flow, so I only touch on the ‘Sign up’ flow to give a wholesome view of my solution. My solution involved introducing a ‘Guest Sign in’ feature while taking advantage of ‘Device Verification’ and ‘Security Questions’.

This solution and implementation covers cases like;

  • Forgotten Passwords
  • Non delivered verification mails
  • Inability to access email used at Sign up

Introducing a Guest Sign In feature

There are legitimate cases where a user no longer has access to the email address used at Sign Up or as in my case doesn’t receive the verification mail. The solution accounts for situations like these thereby reducing drop-offs. The main rationale behind the 'Guest Sign In' was to reduce the 'Effort investments' and make contacting customer care the very last resort. I added a ‘Guest Sign In’ button to the ‘Input Email’ page, on clicking the button, the user is shown a ‘Security Questions’ Page.




Security Questions

Security questions can be tricky and are by no means fool proof, especially in situations where the user’s account holds personal/important details, but they provide an extra layer of security and can be implemented effectively. From my research I was able to distill some of the main factors to consider in offering effective security questions.

  • Answers had to be constant over time
  • Answers had to be specific
  • Questions had to unambiguous etc.
  • Questions couldn’t be generic.

Conclusion

Security is a constantly evolving topic, and this solution is by no means fool proof, but it offers a fine balance between Security and a smooth UX. In implementing this solution, users are encouraged to set security questions at Sign up and the system logs their device. In situations where a user forgets their password, doesn’t have access to email used at Sign Up, doesn’t receive verification mail, the user has the option of using the ‘Guest Sign in’ option to access the account.




Lessons and final thoughts

I learnt a lot about the technicalities behind Sign up flows like hashing and encryption, the project also triggered my thoughts around re-imagining Sign up flows. Because this was a redesign, i was restricted to using the company's brand guidelines and i couldn't really explore layout and visual design ideas.

But most importantly, it reiterated the importance of talking to actual users; initially, because I was solving a problem I encountered, I almost fooled myself into thinking I didn’t need to talk anybody, but talking to the few users I interviewed helped me put things in context. There are more ways to re-imagine this problem, and I will continue exploring.
